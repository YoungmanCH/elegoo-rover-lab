# 構想：自律掃除ロボットの発展ロードマップ（賢い自走・軌跡同期・ウェット拭き・Dyson風可視化・吸引）

> **なぜこの文書があるか**：「自分なりのルンバ」を目指すうえで、やりたい機能（自律走行の高度化／ブラウザへの軌跡同期／クイックル的ウェット拭き／Dyson風の緑色ライト／吸引機構）が **このハード（ELEGOO V4.0 + UNO + ESP32-CAM）で本当に可能か** を、雰囲気ではなく根拠つきで切り分けるため。
> **なんのために**：着手順を決める土台。労力対効果と「諦めるべき所」を先に確定し、デモまでの道筋をブレさせない。
> **だれのために**：実装する自分（と後で読み返す自分）。やる気が出た時に「次どれを掘るか」を即決できるように。
> **位置づけ**：構想・実現可能性評価。**ここに実装コードは置かない**。具体実装は段階md（`docs/project/stageN-*.md`）に落とす。

関連：[code-design.md](../project/code-design.md) ／ [cleaning-logic-spec.md](../project/cleaning-logic-spec.md) ／ [research-route-and-avoidance.md](research-route-and-avoidance.md) ／ [machine-reference.md](machine-reference.md) ／ [stage4-timed-turn.md](../project/stage4-timed-turn.md)

## 0. 結論（先に）

| 区分 | 機能 | 可否 |
|---|---|---|
| すぐ可能（ソフトのみ） | 賢い自律走行（スパイラル/壁伝い/詰まり脱出/超音波スイープ） | ◎ |
| すぐ可能（ソフトのみ） | ブラウザに**推定**軌跡を同期 | ○（真値ではない） |
| 安い後付け（低リスク） | クイックル的ウェットシート引きずり | ◎（機構が単純） |
| 安い後付け（低リスク） | Dyson風 緑色ライトで埃可視化 | ○（要・低出力レーザー＋安全配慮） |
| 中級改造 | ブラシ掃き込み（吸引の代替） | △（3つ目のモータ駆動が要る） |
| 非現実的 | 本物の吸引機構 | ✕（電力・重量・電池がもたない） |

**核心**：「自律的なルンバっぽい動き」と「緑のライト演出」は十分届く。**本物の吸引だけが現実的に無理**。掃除の実体は **ウェット拭き** で取りにいくのが妥当。

---

## 1. 大前提：頭脳がブラウザにある＝賢い処理はUNOに縛られない

本プロジェクトは判断ロジックを **Web層（TypeScript／ブラウザ）** に置き、UNOは「センサを垂れ流し、モータ指令を受ける手足」に徹する設計（[code-design §1.2](../project/code-design.md)）。

- → 地図・軌跡推定・経路計画のような **重い/賢い処理は UNO の RAM 2KB に縛られず**、ブラウザの潤沢な計算資源で書ける。
- → つまり「賢くする」系は**ほぼ全部 `app/` 側の純関数追加**で済み、テストもできる。ハード改造が要るのは物理機構（拭き/光/吸引）だけ。

この切り分けが、本ロードマップ全体の実現可能性を底上げしている。

---

## 2. 最大の制約：ホイールエンコーダが無い（＝真の位置が測れない）

ELEGOO V4.0 の標準駆動モータは**車輪の回転量を測れない**（エンコーダ非搭載）。これが効いてくるのは2点：

1. **真の自己位置(x,y)が出せない** → ブラウザの2Dマップは本質的に「推定」。実機の“本物の軌跡”は描けない（[code-design §6.1](../project/code-design.md) の既存認識と同じ）。
2. **全面カバーを保証できない** → 「床をきっちり網羅」する系統的経路（往復刈り取り）は位置が前提なので不可。

### 2.1 だが悲観しすぎなくていい理由
- 初期型ルンバ自体が **地図を持たず、ランダム＋壁伝い＋スパイラル**で“それっぽく”掃除していた（[research-route-and-avoidance.md](research-route-and-avoidance.md) の整理とも整合）。**今のシンプル方式の延長で「ルンバ感」は十分作れる。**
- 位置が欲しくなったら回避策が2つ：
  - **(A) 推定（dead reckoning）**：指令（前進/旋回）＋ **MPU6050 の yaw**（向き）から姿勢を積分。短時間・小部屋なら見栄えは十分。ドリフトはする。**ソフトのみ・即可能。**
  - **(B) 真値（外部計測）**：天井/真上から **ノートPCのwebカメラ**で車体上のマーカー（ArUco/AprilTag）を追跡 → ブラウザに**本物の軌跡**を描く。ロボット改造ゼロ。中規模の工作。

> 面白い点：旋回制御では捨てた MPU6050 の yaw が、**「軌跡推定」では復活して役立つ**（曲がるためでなく、向きの推定のため）。[stage4-timed-turn.md](../project/stage4-timed-turn.md) でタイマー旋回にした判断はそのままで良い。

## 3. 機能別・実現可能性の詳細

### 3.1 賢い自律走行 ◎（ソフトのみ・最優先）
**やること**：`cleaning.ts` の `step()` 状態機械に「掃除らしい振る舞い」を足す。

- **スパイラル（渦巻き）**：開始時、回転半径を徐々に広げる＝中心から外へ。ルンバの代表挙動。`turnTicks` を時間で減衰させれば作れる。
- **壁伝い（wall-follow）**：壁を一定距離に保って縁を舐める。サーボ超音波を横に向けて距離一定制御。
- **ランダム反射**：壁に当たったら**ランダムな角度**で跳ね返る（今の固定角を乱数化するだけ）。
- **詰まり検知＆脱出**：距離が変化しない/前進してるのに進めてない兆候 → 後退＋大きく旋回の脱出マニューバ。
- **超音波スイープ**：超音波は**サーボ搭載**なので左/中/右に振って測り、**最も空いてる方向**へ曲がる（行き当たりばったりが激減）。

**接続点**：判断は全部 `app/src/domain/cleaning.ts`（純関数）＋ `config.ts`（しきい値）＋ `types.ts`（State拡張）。**Vitestでテスト可能**。UNO側は既存コマンドのみ（駆動 N=3 / 距離 N=21 / サーボ角コマンドは [machine-reference](machine-reference.md) の N 一覧参照）。**新ハード不要。**

**注意**：超音波は**エコー無しのとき `0`（範囲外）**を返す（[machine-reference](machine-reference.md)）。`0` を「壁」と誤判定しない既存ガード（`distanceCm > 0 && < wallCm`）を踏襲すること。

### 3.2 ブラウザへの軌跡同期 ○（推定。真値は外部計測）
- **(A) 推定でいま描く**：`sim/model.ts` を**そのまま「推定器」として流用**。実機の指令と yaw を食わせて pose を進め、`ui.ts` で軌跡として描く。シムと実機で**同じ描画コードが再利用**できる（依存逆転の恩恵）。yaw を使うには firmware から yaw を返すか、UNO で積分して送る経路が要る（要・小改修）。
- **(B) 真値が欲しいなら**：上記 §2.1(B) の真上カメラ＋マーカー。別プロセス（PC側）で追跡し WS でブラウザへ。ロボット非改造。
- **正直な表示**：推定軌跡は「推定」と明示する（実機に位置真値は無い）。デモでは推定で十分“動いてる感”が出る。

### 3.3 クイックル的ウェットシート引きずり ◎（機構が一番簡単）
- **アクチュエータ不要**。底面/後方にシートを軽く床へ押し当てるホルダを付け、走行で引きずるだけ。3Dプリント or 段ボール＋マジックテープで試作可。
- **リスク**：①摩擦増 → モータ負荷↑・電池消費↑。②**カーペットは摩擦で進めない**恐れ（硬い床向き）。③前後重量バランス。
- **検証**：まず**シートを養生テープで底に仮止め**して走らせ、進めるか・拭けるかを見る（タダ）。掃除の“実体”はこれが本命。

### 3.4 Dyson風 緑色ライトで埃可視化 ○（安く映える・要安全配慮）
- 仕組み：Dyson は**緑のラインレーザー**を床すれすれ・極浅角で照射し、微細な埃に長い影を作って見せる。
- 実装：安価な**緑ラインレーザーモジュール（5V・ラインレンズ付き）**を低位置・浅角で固定。**ESP32-CAM の映像に重ねて配信**すれば“埃が見える”絵が撮れ、**デモのインパクト大**。
- **安全**：**低出力（クラス2, <1mW級）**を選び、**人の目に向けない**運用を徹底。資料・発表でも一言注意を入れる。
- 代替：onboard の RGB LED（FastLED/WS2812）や緑LEDフラッドは**安全だが埃可視化効果は弱い**（影を作る浅角がキモなので拡散光では出にくい）。演出だけなら onboard LED を緑点灯でも可。
- **接続点**：レーザー/LEDの ON/OFF は**空きGPIO＋（必要なら）MOSFET/トランジスタ**で。空きピンは**要確認**（[machine-reference](machine-reference.md) のピン割当）。

### 3.5 ブラシ掃き込み（吸引の代替） △（中級改造）
- 小型ギアモータ＋ブラシで debris を掻き込みトレイへ（ルンバのメインブラシ相当）。吸引より遥かに現実的。
- **壁**：駆動の **TB6612 は2chとも車輪で使用済み** → ブラシ用に **3つ目のドライバ or 単方向なら MOSFET 1個＋空きGPIO** が要る。電力・配線・固定の工作が増える。

### 3.6 本物の吸引機構 ✕（非現実的）
- ファン/インペラに**大電流**が必要。**UNO＋7.4V（18650×2）では実用的な吸引力が出ず、電池が保たず、重い**。小型5Vファンは「吸ってる風」止まり。
- 結論：今回は**やらない**。ロマンとして将来課題に温存。掃除実体は §3.3 ウェット拭き（＋余裕あれば §3.5 ブラシ）で取る。

## 4. 推奨ロードマップ（労力対効果の順）

| 段 | 内容 | 種別 | 効果 | リスク |
|---|---|---|---|---|
| 1 | 賢い自律（スパイラル/壁伝い/詰まり脱出/超音波スイープ） | ソフトのみ | 「ルンバ感」最大 | 低 |
| 2 | 推定軌跡をブラウザ描画（`sim/model` を推定器に流用） | ソフトのみ | 見栄え | 低 |
| 3 | 緑ラインレーザー＋カメラ配信 | 安い後付け | Dyson感・デモ映え | 低〜中（安全） |
| 4 | ウェットシート引きずり | 機構（簡単） | “掃除した実体” | 低〜中（床依存） |
| 5 | （任意）真上カメラ＋マーカーで**本物**軌跡 | 別プロセス | 真値の軌跡 | 中 |
| 6 | （任意）ブラシ掃き込み | 中級改造 | 物理回収 | 中 |
| — | 吸引 | — | — | 温存 |

**労力対効果が一番デカいのは ①→③**（ソフトで賢く＋光で映える）。掃除の実体が欲しければ ④ を足す。

## 5. 設計上の指針（既存方針を壊さない）

- **判断は `app/` の純関数に**：新挙動はすべて `cleaning.ts`/`config.ts`/`types.ts` に足し、Vitest でテスト。UNO に判断を入れない（手足原則）。
- **物理アクチュエータの ON/OFF だけ UNO に**：レーザー/ブラシは「指令で点ける/回す」レベルに留め、いつ点けるかの判断はブラウザ側。
- **可視化はシムと実機で描画を共有**：`sim/model.ts` を推定器として再利用し、`ui.ts` の描画を使い回す（依存逆転の利点を活かす）。
- **正直さ**：推定軌跡は「推定」と明示。カバー率は保証しない（[cleaning-logic-spec](../project/cleaning-logic-spec.md) §5 と整合）。

## 6. 未確認・要検証（実機で潰す）

- 空きGPIO の有無（レーザー/ブラシ用）と、MOSFET の要否 → [machine-reference](machine-reference.md) のピン割当で確認。
- yaw を軌跡推定に使うための供給経路（firmware が yaw を返すか／UNO 側で積分して送るか）。
- ウェット拭きを引きずった時の走破性（特にカーペット）と電池持ち。
- USB制御 と WiFi(カメラ) の同時利用の安定性（既出の課題、[code-design §6.1](../project/code-design.md)）。
- 緑レーザーの安全等級と運用ルール（目に向けない／発表時の注意喚起）。

# 構想：自作ルンバの発展ロードマップ

> **なぜこの文書があるか**：「自分なりのルンバ」に向けてやりたい機能（賢い自走／軌跡のブラウザ同期／ウェット拭き／Dyson風の緑ライト／吸引）が、このハード（ELEGOO V4.0 + UNO + ESP32-CAM）で本当に可能かを根拠つきで切り分け、着手順を決めるため。
> **位置づけ**：構想・実現可能性の評価。**実装コードは置かない**。具体化は段階md（`stageN-*.md`）でやる。
> 参照：[code-design.md](../project/code-design.md) ／ [cleaning-logic-spec.md](../project/cleaning-logic-spec.md) ／ [research-route-and-avoidance.md](research-route-and-avoidance.md) ／ [machine-reference.md](machine-reference.md)

---

## 1. 結論（先に）

核心は **「賢い自走」と「緑ライト演出」は十分届く。本物の吸引だけが現実的に無理**。掃除の実体はウェット拭きで取る。

- ◎ **賢い自走**（スパイラル/壁伝い/詰まり脱出/超音波スイープ）― ソフトのみ
- ◎ **ウェットシート引きずり** ― 機構が一番単純
- ○ **軌跡をブラウザに同期** ― ただし「推定」。真値は別途
- ○ **Dyson風 緑ライト** ― 安く映える。ただし安全配慮が要る
- △ **ブラシ掃き込み**（吸引の代替）― モータを1個増やす中級改造
- ✕ **本物の吸引** ― 電力・重量・電池がもたない

労力対効果が一番大きいのは **賢い自走 → 緑ライト**。掃除の実体が欲しければ **ウェット拭き** を足す。

---

## 2. 土台になる2つの事実

この構想全体は、次の「追い風1つ」と「逆風1つ」で決まる。

**追い風：頭脳がブラウザ側にある。**
判断ロジックは Web層（TypeScript）にあり、UNOは「センサを流す・指令を受ける手足」に徹している（[code-design §1.2](../project/code-design.md)）。だから地図・軌跡推定・経路計画のような重い処理は **UNOのRAM 2KBに縛られず**、ブラウザ側で書ける。「賢くする」系はほぼ全部 `app/` の純関数追加で済み、テストもできる。

**逆風：ホイールエンコーダが無い。**
標準モータは車輪の回転量を測れない。つまり **真の自己位置(x,y)が出せない**。影響は2つ：
- ブラウザの2Dマップは本質的に「推定」。実機の“本物の軌跡”は描けない。
- 床をきっちり網羅する系統的経路（往復刈り取り）は不可。

ただし初期型ルンバ自体、地図を持たず **ランダム＋壁伝い＋スパイラル** で掃除していた（[research-route-and-avoidance.md](research-route-and-avoidance.md)）。**今のシンプル方式の延長で「ルンバ感」は十分作れる。**

## 3. やること（おすすめ順）

各項目は「何をする／要るもの・問題／どこを触る」で書く。

### 3.1 賢い自走 ◎（ソフトのみ・最優先）

壁まで直進して旋回するだけの今の挙動に、掃除らしい振る舞いを足す。

- **スパイラル**：回転半径を徐々に広げて中心から外へ。ルンバの代表挙動。
- **壁伝い**：超音波を横へ向け、壁を一定距離に保って縁をなぞる。
- **ランダム反射**：壁に当たったら乱数で曲がる角度を散らす（今の固定角を乱数化）。
- **詰まり脱出**：距離が変化しない＝詰まりと判断し、後退＋大きく旋回。
- **超音波スイープ**：超音波は**サーボ搭載**なので左/中/右に振って測り、一番空いた方向へ曲がる。

**問題**：超音波はエコー無しのとき `0`（範囲外）を返す（[machine-reference](machine-reference.md)）。`0` を壁と誤判定しない既存ガード（`distanceCm > 0 && < wallCm`）を踏襲する。
**触る所**：判断は全部 `app/src/domain/cleaning.ts`（純関数）＋ `config.ts`（しきい値）＋ `types.ts`（State拡張）。UNOは既存コマンドのみ（駆動 N=3／距離 N=21／サーボ角は machine-reference 参照）。新ハード不要。

### 3.2 軌跡をブラウザに同期 ○（推定。真値は別途）

エンコーダが無いので2通りある。

- **推定でいま描く**：`sim/model.ts` を**そのまま「推定器」に流用**し、実機の指令と yaw を食わせて姿勢を進め、`ui.ts` で軌跡を描く。シムと実機で**同じ描画が再利用**できる。短時間・小部屋なら見栄えは十分。ドリフトはする。
- **真値が欲しいなら**：真上から **ノートPCのwebカメラ**で車体のマーカー（ArUco等）を追跡し、ブラウザへ送る。ロボット改造ゼロ・中規模の工作。

**触る所**：推定は `app/` 内で完結（描画は流用）。yaw を使うには firmware が yaw を返すか、UNOで積分して送る経路が要る（小改修）。
**メモ**：旋回制御では捨てた MPU6050 の yaw が、**軌跡推定では復活して役立つ**（曲がるためでなく向きの推定のため）。タイマー旋回の判断（[stage4-timed-turn.md](../project/stage4-timed-turn.md)）はそのままで良い。表示は必ず「推定」と明示する。

### 3.3 ウェットシート引きずり ◎（機構が一番簡単）

底面/後方にシートを軽く床へ押し当てるホルダを付け、走行で引きずるだけ。**アクチュエータ不要**。掃除の“実体”はこれが本命。

**問題**：摩擦増でモータ負荷・電池消費が増える。**カーペットは進めない恐れ**（硬い床向き）。前後の重量バランス。
**検証**：まずシートを養生テープで底に仮止めして走らせ、進めるか・拭けるかを見る（タダ）。

### 3.4 Dyson風 緑ライト ○（安く映える・要安全）

Dyson は**緑のラインレーザー**を床すれすれの極浅角で照射し、微細な埃に長い影を作って見せる。安価な緑ラインレーザーモジュール（5V）を低位置・浅角で固定し、**ESP32-CAM の映像に重ねて配信**すれば“埃が見える”絵になり、デモのインパクトが大きい。

**安全**：**低出力（クラス2, <1mW級）**を選び、**目に向けない**運用を徹底。発表でも一言注意する。
**代替**：onboard の RGB LED や緑フラッド光は**安全だが埃可視化は弱い**（影を作る浅角がキモなので拡散光では出にくい）。演出だけなら onboard LED の緑点灯でも可。
**触る所**：ON/OFF は空きGPIO＋（必要なら）MOSFET。空きピンは**要確認**（machine-reference のピン割当）。

### 3.5 ブラシ掃き込み △（中級改造・吸引の代替）

小型ギアモータ＋ブラシで debris を掻き込みトレイへ（ルンバのメインブラシ相当）。吸引より遥かに現実的。

**問題**：駆動の **TB6612 は2chとも車輪で使用済み**。ブラシ用に3つ目のドライバ、または単方向なら MOSFET 1個＋空きGPIO が要る。配線・固定の工作が増える。

### 3.6 本物の吸引 ✕（非現実的）

ファン/インペラに大電流が要り、**UNO＋7.4V（18650×2）では実用的な吸引力が出ず、電池も保たず、重い**。小型5Vファンは「吸ってる風」止まり。今回はやらず将来課題に温存。掃除実体は 3.3 ウェット拭き（＋余裕あれば 3.5 ブラシ）で取る。

## 4. 設計の指針（既存方針を壊さない）

- **判断は `app/` の純関数に**：新挙動はすべて `cleaning.ts`/`config.ts`/`types.ts` に足し、Vitestでテスト。UNOに判断を入れない（手足原則）。
- **物理アクチュエータは ON/OFF だけ UNO に**：レーザー/ブラシは「点ける/回す」だけ。いつ点けるかの判断はブラウザ側。
- **可視化はシムと実機で描画を共有**：`sim/model.ts` を推定器として再利用し、`ui.ts` の描画を使い回す。
- **正直に**：推定軌跡は「推定」と明示。カバー率は保証しない（[cleaning-logic-spec §5](../project/cleaning-logic-spec.md) と整合）。

## 5. 未確認（実機で潰す）

- 空きGPIOの有無（レーザー/ブラシ用）と MOSFET の要否 → machine-reference のピン割当。
- yaw を軌跡推定に使う供給経路（firmwareが返す／UNOで積分して送る）。
- ウェット拭きの走破性（特にカーペット）と電池持ち。
- USB制御 と WiFi(カメラ) の同時利用の安定性（既出課題、[code-design §6.1](../project/code-design.md)）。
- 緑レーザーの安全等級と運用ルール。

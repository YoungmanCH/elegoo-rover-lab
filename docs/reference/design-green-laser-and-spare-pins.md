## 0. 結論（3行）

1. **物理的に完全フリーなピンはほぼ無い**（"ほぼ全ピンを使い切る構成"）。だが緑レーザーは **ON/OFF 1本** で足りる。
2. 現実的な空きの**筆頭は D11（サーボY）**。標準車はサーボ1個（Z）だけで、**Y用サーボヘッダ（信号D11 ＋5V ＋GND）が丸ごと空く**。
3. **GPIO から直接レーザーは焚けない** → どのピンを選んでも **MOSFET（or トランジスタ）＋5Vレール給電** が必須。ピンは「制御信号」だけを担う。

## 2. 大前提：GPIO で直接レーザーは焚けない

- **ATmega328P のピン電流**：推奨 ~20mA、絶対最大 40mA/ピン（さらにポート/チップ合計の上限あり）。
- **緑ラインレーザーモジュール（5V品）**：典型 **数十〜150mA級**。→ **ピンから直接給電は不可／危険**。
- したがって構成は必ず：

```
   UNO GPIO ──[ゲート抵抗]── G  NチャネルMOSFET
                              D ── レーザー(−)
   レーザー(+) ── +5V(レール)  S ── GND
                            （G–S間に 10kΩ プルダウン）
```

- **ピンに求められるのは「ゲートを叩く信号」だけ** ＝ 電流は問題にならない ＝ **空いてさえいればどのGPIOでもよい**。これが「D11 で十分」と言える根拠。
- 部品例：ロジックレベルMOSFET（**AO3400 / 2N7000 / IRLZ44N** など）＋ゲート抵抗 100〜220Ω＋プルダウン 10kΩ。
- ソフト制御が不要なら 5V 直結で常時点灯も可だが、roadmap 方針（[§「物理アクチュエータはON/OFFだけUNOに」]）に沿って**ソフトON/OFF**にするのが本筋。

## 3. 推奨：D11（サーボYヘッダ）— 根拠と実コード

### なぜ D11 が筆頭か
1. **物理的に本当に空く**：標準 V4.0 は超音波を載せた **Zサーボ1個のみ**。Yサーボ（垂直チルト）は未搭載が普通で、シールド上の **Y用3ピンサーボヘッダ（信号D11／＋5V／GND）が未使用**。→ **レーザー＋MOSFETに要る 信号・5V・GND が1コネクタで揃う**。配線が最もきれい。
2. **頭脳が触らない**：app(TypeScript) はサーボYに一切コマンドを送らない（掃除ループは N=5 D1=2 を出さない）。
3. **通常時 D11 はフロート**：ファームの単一 `Servo myservo` オブジェクトは、Init で Z→Y を順に当てた後 **`detach()` する**（[FW-cpp:337-341]）。以降 D11 にパルスが出るのは `Servo_controls(Servo=2 or 3)` が呼ばれた時だけ（[FW-cpp:406-420]）＝Y/カメラ縦チルト指令時のみ。普段は誰も使っていない。

```cpp
// DeviceDriverSet_xxx0.cpp  Servo_Init() より（出典 [FW-cpp:330-342]）
myservo.attach(PIN_Servo_z, ...); myservo.write(90); delay_xxx(500);
myservo.attach(PIN_Servo_y, ...); myservo.write(90); delay_xxx(500);  // ← 起動時に一瞬だけD11にパルス
myservo.detach();                                                     // ← 以降D11は解放（フロート）
```

### 注意（このまま使うと起きること）
- **起動時に一瞬だけ D11 へサーボパルスが出る**（上記 Init の Y attach）。MOSFET を介すと **ブート時にレーザーが一瞬光る**。実害は小さいが、気になるなら下記の firmware 変更で消す。
- D11 を `digitalWrite` の ON/OFF 線として使うなら、**Yサーボ系のコード経路と排他**にすること（同じピンに Servo パルスと digitalWrite が混在しないように）。

## 4. 次点・非推奨の選択肢

| 候補 | 評価 | 理由 |
|---|---|---|
| **D9（IR受信）** | △ 次点 | ブラウザ/シリアル制御の現状、IRリモコンは未使用なので論理的には空く。だが IR受信モジュールが**物理接続されており、外す/付け替えの手間**がある分 D11 に劣る。 |
| **A0〜A2（ライン×3）** | △ 温存推奨 | 頭脳は未使用だが、roadmap で**崖センサ転用の余地**として残す資産。レーザーのために潰すのは惜しい。アナログピンは `digitalWrite` 可（A0=D14…）だが、ライン基板コネクタが物理占有。 |
| **A3（電圧）** | ❌ | 配線済で将来の残量可視化に使う。転用しない。 |
| **D4（RGB LED）流用** | ─ 別案 | 新ピン不要で**緑“点灯”だけ**ならオンボード RGB を緑にするだけ。ただし**拡散光なので埃可視化は弱い**（影を作る浅角がキモ＝ラインレーザーの代替にはならない）。演出のみの妥協案（roadmap §3.4 代替）。 |

## 5. firmware 変更チェックリスト（D11採用時）

対象は自作フォーク `arduino/SmartRobotCarV4.0_DIY/`。手足側は **ON/OFF だけ**持たせる（点けるタイミングの判断はブラウザ）。

1. **Yサーボ経路を無効化**（D11をレーザー専用に）
   - `Servo_Init()` の Y attach 2行（[FW-cpp:337-338]）を削除 → **ブート時の一瞬点灯を消す**。
   - `Servo_controls()` の `Servo==2/3` 分岐（[FW-cpp:406-420]）を使わない／削る。
2. **D11 を出力初期化**：`pinMode(11, OUTPUT); digitalWrite(11, LOW);`（setup で）。
3. **レーザー用コマンドを1個追加**（独自JSONプロトコルに増設）。例：
   - `N=30, D1=0/1` → `digitalWrite(11, D1 ? HIGH : LOW)` ＋ `{ID_ok}` を返す。
   - ※ N番号は既存（1,3,4,5,21,22,23,100,101,102,104,106,110）と衝突しない値を選ぶ。30 は空き。
4. **安全弁との整合**：緊急停止（`{"N":100}` / stop）時に**レーザーも消す**ようにする（離席・全停止でレーザーが点きっぱなしを防ぐ）。

> app(TypeScript)側は、roadmap の「ON/OFFだけUNOに」方針通り、`encodeLaser(on)` 純関数＋UIトグルを足すだけ。掃除ループ本体は不変。

## 6. 安全（緑レーザーは"映え"と引き換えにリスク）

- **直視厳禁**。緑（515〜532nm）は視感度が高く、同出力でも赤より眩しく網膜リスクが高い。
- **低出力品（クラス1〜2相当、<1mW目安）を選ぶ**。高出力ラインレーザーは避ける。
- **床すれすれ・極浅角で固定**し、ビームが**水平〜上向きに漏れない**マウントにする（浅角固定は埃可視化の原理上も正解）。
- ペット・子どもの目線に入らない高さ・向き。デモ時は人に向けない運用ルールを決める。
- roadmap §3.4 の未確認事項「**緑レーザーの安全等級と運用ルール**」は、購入モジュールのクラス表記を実機で確認して本書に追記すること。

# 実機シリアル通信デバッグ：仮説と切り分け（段階3）

> **目的**：シリアルモニタでは応答するのに、ブラウザ（Web Serial / `transport`）経由だと `read()` が `serial timeout` になる原因を、仮説→検証で特定する。
> **方針（重要）**：**コード変更・診断ログ追加・パラメータ変更は、すべて事前承認を得てから行う。** 本書は「何をどう確かめるか」の計画であって、まだ何も変更しない。

---

## 1. 状況
- **シリアルモニタ（Arduino IDE）**：`{"H":"21","N":21,"D1":2}` → `{21_0}` が返る＝ファームはJSONプロトコルを喋る。
- **ブラウザ（transport経由）**：`read()` が `serial timeout`。yaw を `catch(()=>0)` にしても改善せず＝**距離(N=21)の段階で落ちている**（yawの手前）。
- 付随：`実機接続`を2回押すと `The port is already open`。起動時に `MPU6050_chip_id: 0`（ジャイロ未検出）。

## 2. 確定事実（観測）
| # | 観測 | 意味 |
|---|---|---|
| O1 | Serial Monitor で `{21_0}` が返る | ファームは N=21 に正しく応答する |
| O2 | ブラウザの read が `serial timeout` | transport 経由だと応答を受け取れていない |
| O3 | yaw を catch しても失敗 | 距離(N=21)の段階で落ちている |
| O4 | `already open`（2回押し） | 二重 open。ポートは1接続のみ |
| O5 | `MPU6050_chip_id: 0` | ジャイロ未検出（yaw/N=24に影響。今回の timeout とは別問題） |
| O6 | **車が動いたか未確認** | 送信(N=3)が届くかの切り分けに必須（最重要の欠けている情報） |

---

## 3. 仮説（ランク順）

### H1（最有力）接続時の Arduino 自動リセットで受信が成立していない
- Web Serial の `port.open()` は DTR をトグルし、**Arduino をリセット**する。
- 現コードは `_readLoop()` を**開始してから**2秒待つ（`transport.ts` の `open()`）。リセットの瞬間に readable ストリームが中断/エラーになると `reader.read()` が reject → for ループを抜けて `_readLoop` が終了 → **以後一切受信しない** → 全 read が timeout。
- さらに、リセット直後はファームが `setup()` 中（`chip_id:0` を複数回出すなど初期化に時間）で、2秒では `loop()` に未到達 → 最初の問い合わせに無応答の可能性。
- **検証**：受信生データを一時的にログ（RX）。
  - RX が全く出ない → 受信ループが死/未受信（H1 濃厚）。
  - RX に起動ノイズは出るが `{21_…}` が来ない → ファームがブラウザ経由の問い合わせに応答していない（タイミング/モード）。
  - RX に `{21_0}` が出るのに read 失敗 → 受信はできておりパース/対応付けの問題（→ H6）。
- **候補対処（承認後）**：受信ループをリセット待ちの**後**に開始／受信ループをエラー耐性化（落ちても再起動）／リセット待ちを延長。

### H2 タイムアウト 300ms が短い
- ファームは受信フレームを**エコー**（`Serial.println`）し、超音波 ping(~30ms)＋9600baud 往復＋loop 周期で、状況により 300ms を超え得る。
- 単独では弱い（Serial Monitor は即返って見えた）が、H1 と複合で効く。
- **検証**：延長で改善するか。**候補対処**：300 → 1000〜1500ms。

### H3 連続出力でバッファ/パースが乱れる
- `MPU6050_chip_id: 0` が loop で繰り返し出ていると、`{}` を含まないテキストが大量に来る。`_extractFrames` は `{}` 無しだと buffer をトリムしない → 膨張。`{21_0}` 到着で抽出自体はできるはずだが、極端だと遅延の一因。
- **検証**：RX ログで出力頻度を見る。**候補対処**：影響は限定的、優先度中〜低。

### H4 送信(write)が届いていない
- コマンド自体が実機に届いていなければ応答も無い → timeout。
- **検証＝O6**：**接続時に車が前進したか**。動けば送信は OK ＝ 受信側(H1)が濃厚。動かなければ送信も要調査。

### H5 二重 open（ポート状態）
- 2回押しで既存接続が残り状態が壊れる。リロード/USB 挿し直しで解消。
- **候補対処**：ボタン二重起動ガード／`already open` のハンドリング。今回の timeout の主因ではないが要整理。

### H6 エコー/フレーム対応付け（低）
- ファームのエコー `{"H":"21",...}` は `_` を含まず `parseFrame`→null→skip。query ループが読み飛ばすので正常。ただしエコー1個分を消費するので、短い timeout だと余裕が減る（→ H2 と関連）。

## 4. 切り分け手順（最小の診断から／コード変更は承認後）
1. **O6 を確認**：接続時に**車が前進→停止したか**。送信(N=3)の生死を判定する（受信問題か送信問題かの分岐）。
2. **RX ログを一時追加（承認制）**：受信生データを出して O2 の所在を特定する。
   - 「RX 全く無し」／「起動ノイズのみ」／「`{21_0}` まで来る」のどれか、で次が決まる。
3. 結果に応じて**最小の対処を1つずつ**適用（有力順：H1 リセット順 → H2 タイムアウト）。各変更は事前承認のうえで。

## 5. 今ほしい2つの情報（これで受信/送信を確定）
- **A. 車は前進→停止したか？**（H4 の判定）
- **B. RX ログに何が出るか**（H1〜H3 の判定）。※B はログ追加の承認をもらってから。

## 6. 結論（実機で確定）
RX/TX ログを入れて確定した。**Web 側は完全に無罪**（`[TX]` も `[RX]` も正常に出ていた）。原因は **Arduino firmware**。

### 真因：`MPU6050_getdata.cpp` の `MPU6050_dveInit()` が `setup()` を抜けない
- `uint8_t cout;` が**未初期化**のまま `cout += 1; if (cout > 10) return true;` に使われていた（未定義動作）。「10回試して諦める」脱出が**信頼できず発火しない**ビルドになっていた。
- そのため gyro が `chip_id == 0`（I2C未ACK）を返す間、`do/while` を**永久に回し続ける** → `setup()` を抜けず `loop()` に到達しない → N=21/N=3/N=4 すべて無反応（read timeout・車も動かない）。
- ログの「`MPU6050_chip_id: 0` が無限連続・`{...}` フレームゼロ」がこれ。

### 対処（適用済み）
`MPU6050_dveInit()` を**必ず終わる有界 for ループ**に書き換え：
```cpp
for (uint8_t cout = 0; cout < 10; cout++) {       // 0で初期化・最大10回で必ず抜ける
  chip_id = accelgyro.getDeviceID();
  Serial.print("MPU6050_chip_id: "); Serial.println(chip_id);
  if (chip_id != 0x00 && chip_id != 0xFF) { accelgyro.initialize(); return false; }
  delay(10);
}
return true;   // gyro 無しでも諦めて loop() へ進む(yaw は使わない)
```
- バックアップ：`MPU6050_getdata.cpp.orig`（同フォルダ。戻すときは `.orig`→`.cpp`）。
- 焼く対象は **Downloads** 配下の実体：`…/08 …/SmartRobotCarV4.0_V1_20220303/`。

### 結果
- `chip_id` が**約7〜10回で停止** → `{21_0}` が返る → **N=3 で車が前進した**（送受信・駆動すべて生存を確認）。

## 7. 残課題：Web 側のタイミング（次に直す）
firmware は正常になったが、`read()` がまだ落ちる。原因は3つ：
1. **起動と read の競合**：`port.open()` が Arduino をリセットし、起動中（chip_id 出力中）に read を撃って間に合わない。
2. **off-by-one**：`TIMEOUT_MS=300` が短く、各 query が自分の応答を待ちきれず、遅れて届いた `{21_0}` を次の query が拾う。
3. **`error:deserializeJson`**：timeout で応答を待たず連射するので firmware 受信バッファでフレームが混ざり、たまにパース失敗（我々のパーサ的には実害小）。

### 予定の対処（承認後に適用）
- `serial-robot.ts`：`TIMEOUT_MS` `300` → `1500`（各 query が自分の応答を待てる＝off-by-one 解消）。
- `transport.ts` の `open()`：リセット待ちの後、**起動ノイズを破棄**してから返す（`frames`/`buffer` をクリア）。

## 8. 物理的制約：USB テザー（別件・要検討）
- アップロードもデータ通信も **USB ケーブル**経由。モーター電源は車載バッテリだが、**シリアル通信の線（USB）が短く**、机の上でしか動けない。
- 候補：(A) **アクティブUSB延長ケーブル**で半径を伸ばす（最速・現アーキテクチャ維持）／(B) **ESP32-CAM の WiFi 経由**（本来の無線。ただしブラウザは生TCP不可なので WebSocket化 or ローカル中継が必要）／(C) **自走ロジックを基板側(C++)に載せて完全自律**（無線不要だが移植が大きい）。
